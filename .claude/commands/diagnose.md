# /diagnose — エラーの多角的分析

$ARGUMENTS にエラーを受け取り、修正前に以下6視点で分析する。
分析完了まで修正コードを書かないこと。

## 分析フレームワーク
1. 🔍 直接原因（What）— エラーの意味、該当ファイル・行
2. 🧬 根本原因（Why）— なぜこのコードが書かれたか、設計のズレ
3. 💥 影響範囲（Blast Radius）— 他に同じパターンがないかgrep検索、ユーザー影響
4. 🔐 セキュリティ観点 — 悪用可能性、認証バイパス、データ漏洩
5. 🔄 再発防止 — テスト追加、ルール追加、型制約
6. 🛠️ 修正計画 — パッチ vs リファクタ、推奨と手順

同一パターンの検索は必ず実行すること。
分析結果を提示し、承認を得てから修正に着手する。
